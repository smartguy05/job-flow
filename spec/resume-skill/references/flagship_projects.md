# Selecting and Presenting Projects

Projects come from the **user's career info** — never invent them. Use this guide to decide
which documented projects to include and how to describe them for a given role.

## Which projects to include
- Include **2–4 projects**, chosen for relevance to the target role.
- Lead with the user's strongest, most impressive work (a "flagship" project) — the one
  that best demonstrates depth, ownership, or impact.
- Prefer projects whose domain, stack, or problem overlaps the job description.
- Drop projects that add length without adding relevance.
- If the career info documents no meaningful projects, omit the section entirely.

## Matching projects to roles
Read each documented project for its signals, then match:
- **Security / infrastructure roles** → projects involving monitoring, detection, hardening,
  networking, or reliability.
- **AI / ML roles** → projects using LLMs, agents, RAG, tool use, or model integration.
- **Developer-tools roles** → CLIs, IDE extensions, SDKs, automation.
- **Data roles** → databases, pipelines, query tooling, analytics.
- **Frontend / full-stack roles** → apps with notable UI, performance, or product scope.

## How to describe a project
- 15–30 words. Lead with what it *does* and the aspect most relevant to the target role.
- Note the primary technologies in parentheses after the name.
- Emphasize the facets that matter for this job; de-emphasize the rest.
- Everything stated must be verifiable from the career info.

## Pointing to more work
If the user has a portfolio or public profile (e.g. GitHub) with many more repositories,
close the section with a short line pointing to it — using the link and any repo count
found in the career info.
